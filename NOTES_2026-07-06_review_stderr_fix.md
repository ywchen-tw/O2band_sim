## 2. MC-stderr (ddof) correction — applied

### Problem

`mca_out_lbl` computed the run-to-run spread with `np.std(r, axis=0)`
(population convention, ddof=0) over the **Nrun = 3** independent MCARaTS
runs.  The unbiased sample std (ddof=1) is larger by exactly
√(Nrun/(Nrun−1)) = √1.5 ≈ **1.224745**, so all delivered
`*_std` / `*_stderr` datasets were ~18% low.  Means (reflectance, radiance,
OT) were never affected — the correction is an exact algebraic rescale of the
error bars only.

### Code changes

| file | change |
|---|---|
| `src/util/mca_out_lbl.py` | `r_raw_std` now uses `np.std(..., ddof=1)` (zeros fallback at Nrun = 1) |
| `src/sim_o2band.py` | `_write_chunk` stamps `stderr_ddof = 1` on every new chunk; `_assemble_band` rescales any **unmarked** (legacy) chunk on the fly by √(Nrun/(Nrun−1)) using the chunk's own `Nrun` attr; `_write_band_payload` stamps assembled files/groups |
| `src/fix_stderr_ddof.py` | **new** — idempotent in-place fixer for existing files: rescales the four std/stderr datasets (chunk files) or the two stderr datasets (assembled per-band / merged files), stamps `stderr_ddof = 1`, skips marked files, atomic tmp+`os.replace`, `--dry-run` supported |

### Data corrected (in place, marker `stderr_ddof = 1`)

- `/scratch/alpine/yuch8913/O2band_sim/z120_p1e6_n3` (delivered production):
  **195/195 files** — 192 chunks + `o2a.h5` + `o2b.h5` + `o2band_benchmark.h5`.
- `/scratch/alpine/yuch8913/O2band_sim/z120_p1e7_n3` (P=1e7 run, in progress):
  **187 existing chunks** fixed; the stale partial `o2band_benchmark.h5` had
  nothing to fix and will be regenerated by the queued assemble job.

### Verification

- On a held-aside chunk copy: only `ref_std`, `ref_stderr`, `rad_std`,
  `rad_stderr` changed (exactly ×1.224745); `wvl`, `f0`, `idx`, `r_raw`,
  `ref`, `rad` **bit-identical**; re-running the fixer is a no-op.
- Assembled `o2a.h5` cross-checked against its corrected chunks after the fix
  (reflectance, radiance, stderr all exactly consistent; marker present).
- `noise_report.py` on the corrected merged file: worst relative-stderr p95
  = **0.0057** → still **PASS** vs the 0.01 gate.  Corrected figures:
  median relative stderr ~6×10⁻⁴; p95 ≤ 5.7×10⁻³ (O2A) / ≤ 1.6×10⁻³ (O2B).

### Concurrent-run safety (P=1e7 array on blanca)

Array job 26863596 (`o2b_run`) was **still running** during the fix.  Tasks
already running keep the *old* code in memory and write unmarked ddof=0
chunks; tasks that start/requeue after the edit write marked ddof=1 chunks.
Both are handled: the correction-aware `assemble` (new code, loaded when the
pending assemble job starts) rescales unmarked chunks on the fly, so the
final HDF5s are correct regardless of the mix.  Re-running
`fix_stderr_ddof.py` after the array drains is optional tidying of the chunk
files themselves, not required for the products.
